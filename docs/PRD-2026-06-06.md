## 1. Executive summary

makeLILA is the unified internal operations platform for VCycene's LILA Composter business. It replaces what the market would otherwise stitch together as five separate SaaS subscriptions — a CRM, a customer service desk, a lightweight ERP, a finance/lifecycle marketing stack, and a pre-sale campaign manager — with one React + TypeScript + Supabase application owned by the team and built around hardware-product specifics that off-the-shelf tools do not model natively (serial-tracked units, telemetry-classified machine state, batch genealogy, depot-style replacement workflows).

The strategic posture, ratified by the [2026-06-06 competitive landscape analysis](competitive-landscape-and-proposal-2026-06-06.md), is that **makeLILA is the system of record for internal operational data; every external platform is either an input that seeds records or an event sink that receives state changes**. This PRD documents the product as it exists today across nine functional modules, articulates the unified five-domain vision, locks in integration roles for Shopify / HubSpot / Klaviyo / Facebook / Google Workspace / Quo, and prioritizes the roadmap into P1 / P2 / P3 tiers aligned to the competitive proposal.

The single biggest open product decision captured here is whether HubSpot remains the upstream CRM (lead capture, marketing automation tier-1, sequences) or is decommissioned in favor of a thinner Facebook-Ads + Klaviyo + makeLILA stack with HubSpot's CRM responsibilities migrated in-house. The decision framework is in §6.

## 2. Vision: a five-domain unified platform

The five operational domains makeLILA must serve, with the strategic question for each:

**Pre-sale marketing & campaigns.** Today this lives entirely outside makeLILA — Facebook Ads, Klaviyo email broadcasts, HubSpot sequences. The product gap: campaign performance, lead attribution, and cost-per-acquisition are not visible inside makeLILA, so operators cannot tie a fulfilled order back to the campaign that originated it. The vision is to ingest Facebook Ads spend + lead form submissions + Klaviyo campaign events into a single `marketing_attribution` view per customer.

**CRM (lead → customer lifecycle).** HubSpot owns this surface today and is the system of record for pre-sale contacts. makeLILA's [Customers](app/src/modules/Customers) module is the system of record for post-sale customer state, with the [JourneyTab](app/src/modules/Customers/JourneyTab.tsx) modeling lifecycle stages (consideration → onboarding → routine_use → at_risk → churned). The vision is to make this continuous: a HubSpot contact becomes a makeLILA customer at order-creation, and the journey stage continues to evolve through telemetry and ticket signals.

**Service (support, repair, replacement).** Fully owned by makeLILA via the [Service](app/src/modules/Service) module — Inbox (Gmail-synced), Support Tickets, Onboarding, Replacement, with attachments, conversation threading, classifier metadata, and Calendly-booked diagnosis calls. The vision is to make every ticket device-aware (firmware version, latest telemetry classification, repair history surfaced in a header) and to auto-create tickets from persistent non-OK telemetry states.

**ERP (orders, inventory, fulfillment, build, returns).** Fully owned by makeLILA via [OrderReview](app/src/modules/OrderReview), [Fulfillment](app/src/modules/Fulfillment), [Build](app/src/modules/Build), [Stock](app/src/modules/Stock), and [PostShipment](app/src/modules/PostShipment). The vision is to finish the Returns/Refunds overhaul (grade + disposition + responsible team + quarantine), promote build station passes from implicit columns to event rows, and persist freight quote history.

**Lifecycle marketing & finance.** Partially owned. Klaviyo handles broadcast email but is wired one-way (Shopify → Klaviyo, not makeLILA → Klaviyo). QuickBooks Online is reconciled manually each Monday. The vision is a Klaviyo Track API firehose for every internal state change (`unit_shipped`, `service_ticket_resolved`, `replacement_shipped`), a daily-summary QBO journal sync, and a finance-review approval gate inside the active Returns overhaul.

## 4. Functional surface area — module by module

The nine modules and how each maps to the five-domain vision. Italicized items are planned (per competitive proposal) and not yet shipped.

### 4.1 [OrderReview](app/src/modules/OrderReview) — ERP

Source-of-truth for incoming sales orders synced from Shopify. Operators triage by tab (Pending / Out / Flagged / Confirmed / Replacement / All). Per-order detail panel shows AddressCard (Google Maps + Claude LLM verification), LineItemsCard, ReadinessChecklist (contact + address checks; freight check dropped 2026-06-04 per Pedrum policy), FreightCard (ClickShip / Freightcom). The Replacement tab houses `kind='replacement'` orders including the `awaiting_batch_id` queue (e.g. Kristen Pimentel R-0001 + Cheryl Lemieux R-0032 awaiting P100X).

*Planned: PaymentCard surfacing Shopify subtotal/tax/discount/payment-method columns already on `orders` (backlog #4 substrate shipped 2026-05-26); freight quote history child table preserving every ClickShip/Freightcom re-quote (backlog #11 in proposal).*

### 4.2 [Fulfillment](app/src/modules/Fulfillment) — ERP

Six-step queue (assign serial → test → dock → label → email → fulfilled) plus a Shelf view tracking 25-unit skids and a History tab showing both in-app fulfilled units and the 157-row Excel-imported `fulfillment_log`. Carrier enum expanded to UPS / FedEx / Purolator / Canada Post / Canpar / GLS as of 2026-06-05.

### 4.3 [Build](app/src/modules/Build) — ERP

Pipeline board for production batches (P50 retired, P150, P50N, P100, P100X in-production). Per-unit detail includes firmware version, technician, electrical/mechanical check status, defect notes. *Planned: promote per-unit QC columns into a `build_station_passes` event-row table so rework attempts, station-level photos, and defect categories per station become first-class queryable; ships a Build QC Dashboard (defects by station, by technician, first-pass yield trend) — Tulip MES pattern, backlog #9 in proposal.*

### 4.4 [PostShipment](app/src/modules/PostShipment) — ERP + Finance

Returns, Refunds, Replacements, Cancellations, History tabs. Active overhaul (P1 backlog #2): `refund_method` enum (original Shopify card / Sezzle / QuickBooks CC / bank e-transfer / store credit), `finance_review` FSM, approver gates via `MANAGER_EMAILS` / `FINANCE_EMAILS`, refund dashboard with responsible-team / channel / unit-condition / monthly-trend tiles. *Planned: extend `returns` with `grade` (A_resale / B_refurbish / C_secondary / D_scrap), `disposition` (restock / refurbish / RTV / scrap / parts_harvest), `responsible_team`, `inspection_findings`, `inspection_photos`, `recoverable_value_usd`; add a quarantine `units.status` value so received-but-uninspected units do not re-enter Available-to-Promise — closes backlog #2 + #79 together.*

### 4.5 [Service](app/src/modules/Service) — Service domain

Inbox tab consumes Gmail via the sync edge function, classifier triages incoming threads. Support Tickets tab provides the full ticket FSM (open / waiting_on_us / waiting_on_customer / closed), priority, owner, notes audit feed with attached system events (📎 Added photo, 🗑 Removed photo), Calendly-booked diagnosis links, classification log. Onboarding tab tracks customer onboarding state. Replacement tab manages the replacement workflow including the awaiting-batch queue.

*Planned: device-context header on every ticket (firmware + QC + telemetry classification + repair history chips) — Salesforce Field Service / Aquant pattern; SLA aging + auto-escalation (Zendesk pattern); auto-create tickets from persistent non-OK telemetry states — Aquant / Peloton pattern; AI-drafted reply suggestions grounded in the Templates corpus — Intercom Fin pattern.*

### 4.6 [Stock](app/src/modules/Stock) — ERP

Units (serial-level with status enum, batch FK, customer FK via #67), Parts inventory, Batch inventory. Backlog #71 added `orders.awaiting_batch_id` so replacement orders queued for a future inbound batch (P100X end-of-July 2026) are first-class.

### 4.7 [Customers](app/src/modules/Customers) — CRM

Directory + JourneyTab modeling lifecycle stages. JourneyTab includes a manual stage override (`journey_stage_override`) so operators can correct inference. *Planned: right-side widgets sidebar on customer detail (open tickets, in-flight returns/replacements, last 3 orders, linked machines, predictive churn risk) — Attio / Vitally record-page pattern; pull-back of Klaviyo predictive properties (EDNO, predicted CLV, churn risk) as nightly cron — Klaviyo predictive analytics pattern.*

### 4.8 [Templates](app/src/modules/Templates) — supporting layer

Editable email/SMS templates with variable interpolation. Used by status SMS sends from Dashboard, by ticket replies, by Order Review and PostShipment flows.

### 4.9 [ActivityLog](app/src/modules/ActivityLog) — supporting layer + OKR/KPI surface

Audit trail backed by `activity_log` table. Every mutation in every `lib/*.ts` module calls `logAction(action_type, ref_id, detail)`. KPI panel surfaces daily / 7d / 30d counts per action type. *Planned: add `entity_type` / `entity_id` / `unit_serial` columns so "show me everything that ever happened to serial X" is a single index query rather than text-pattern matching — backlog #12 in proposal.*

**Planned: OKR & KPI tracking — company-level + individual-level.** The activity_log is already a high-fidelity record of every interaction every team member has with makeLILA. Promoting it from "audit log" to "OKR substrate" turns goal tracking into a passive byproduct of doing the work, not a manual spreadsheet exercise.

- **Data model.** New `okrs` table: `id`, `scope` enum (`company` / `team` / `individual`), `owner_id` (FK `profiles.id`, null when scope = company), `name`, `description`, `target_value numeric`, `target_unit text` (e.g. `units`, `tickets`, `CAD`, `percent`), `period_start`, `period_end` (typically quarterly), `source_action_types text[]` (which `activity_log.type` values aggregate into this OKR), `aggregation` enum (`count` / `sum_detail_numeric` / `distinct_ref_id`), `manual_current_value numeric` (for OKRs not derivable from activity_log e.g. NPS), `created_by`, `created_at`.
- **Computed view.** A `okr_progress` view joins each OKR to a live aggregate over `activity_log` rows filtered by `(action_type IN source_action_types) AND (created_at BETWEEN period_start AND period_end) AND (user_id = owner_id when individual)`. Returns `current_value`, `pct_complete`, `pace` (linear-extrapolated EOY/EOQ projection), `on_track bool`.
- **Company OKRs (examples).** Q3 2026 ship 200 units (source: `action_type IN ('fulfillment_completed','unit_shipped')`); resolve 95% of P1 tickets within 24h SLA (sources: ticket FSM transitions); revenue 250k CAD (source: `qbo_daily_journals` aggregate, manual fallback); net churn under 5%.
- **Individual KPIs (examples derived from existing action_types).** Raymond: `fulfillment_completed` count, `unit_qc_passed` count per week, average days-in-queue. Pedrum: `order_approved` count, `address_verified` count, average time-to-confirm. Junaid: `ticket_resolved` count, average first-response-minutes, ticket_resolution_minutes. George: `refund_approved` count, refund-approval median latency. Reina: `onboarding_completed` count, `customer_followup_sent` count. Huayi: `telemetry_status_changed` reviews, backlog item completion velocity.
- **UI surface.** A new `OKRsTab.tsx` next to the existing KPI panel inside the ActivityLog module. Two views: **Company OKRs** (cards with progress bars, pace projection, on-track / behind / ahead chip) and **My OKRs / KPIs** (operator-scoped, default to current user, filterable by team member for managers). Each card click drilldowns to the underlying `activity_log` rows powering the count — the audit trail and the goal tracker stay one click apart.
- **Manual vs derived.** Most OKRs derive from `activity_log` directly. A few (NPS, customer satisfaction, marketing CPA, revenue when QBO is mid-month) need a `manual_current_value` field updated by the owner. The view coalesces derived → manual → null.
- **Privacy + access.** Individual KPI views are visible to (a) the individual themselves, (b) their manager (via `profiles.role`), and (c) Huayi + George. No public-to-team view of individual scoring until that decision is explicitly opted into by the team — protects against the OKR-as-surveillance failure mode.
- **Touches.** New [lib/okrs.ts](app/src/lib/okrs.ts), new [modules/ActivityLog/OKRsTab.tsx](app/src/modules/ActivityLog), migration `okrs_table` + `okr_progress_view`, seed file for the initial Q3 2026 company OKRs.
- **Effort.** M. **Sequencing.** Depends on proposal P2 #12 (activity_log entity refs) shipping first so KPIs can be cleanly attributed by user_id without text-pattern matching `detail`. Then OKRs become a thin SQL view on top.

### 4.11 Finance (NEW, visibility-restricted to George + Huayi) — Finance domain

A new dedicated top-level module for forward-looking finance work. **Hidden by default in the GlobalNav for everyone except George and Huayi** per the RBAC enhancement in §8 (extending the `canDo` helper with `canView(role, module)`). Restriction is enforced at three layers: navigation entry hidden client-side, route guard redirects unauthorized users, and Supabase RLS denies SELECT on the underlying tables to non-finance roles.

This module absorbs and extends the planned `PostShipment/FinanceTab` (QBO daily-summary journal sync from §6.15) and adds two new forward-looking surfaces — real-time **sales projection** and real-time **production projection** — that together give George and Huayi a single screen for "where are we tracking financially this quarter."

#### 4.11.1 QBO daily-summary journal panel

Moved out of PostShipment into this restricted module. Same scope as documented in §6.15: nightly aggregation, one journal per `(currency, payment_channel)` group, repost + view-in-QBO buttons, exception rows for failures. Lives here because the GL detail is finance-only.

#### 4.11.2 Real-time sales projection

A live forecast view of incoming revenue, refreshed on every order / refund / payment event via the existing realtime subscriptions.

- **Inputs:** trailing 30/60/90-day order velocity from `orders`, average order value by SKU and batch, the marketing-attribution signals from §5 (Facebook campaign performance + Klaviyo conversions + HubSpot pipeline), and a seasonality multiplier (a manual `seasonality_curve` JSON owned by Huayi).
- **Model.** Two side-by-side projections — a **rolling-average forecast** (last-90d order velocity × AOV by SKU × seasonality) and a **pipeline-driven forecast** (Facebook lead-ads in-flight × historical lead-to-order conversion + active HubSpot sequences × historical conversion). The delta between them is the read on whether marketing is over-delivering or under-delivering relative to baseline.
- **UI.** A `SalesProjectionPanel.tsx` showing: a chart with historical revenue (solid) + projected next-30/60/90-day revenue (shaded confidence band, 80% interval); KPI tiles for projected MRR-equivalent / quarter-to-date / quarter-end estimate vs the company revenue OKR; breakdown table by batch (P150 / P50N / P100 / P100X) so "P100X stock is the bottleneck for September revenue" is visible. Auto-flags when projection crosses below the company revenue OKR pace.
- **Data model.** New table `sales_projection_snapshots` (`id`, `as_of`, `horizon_days`, `model` enum `rolling_avg` / `pipeline_driven`, `projected_revenue_cad`, `projected_revenue_usd`, `lower_bound`, `upper_bound`, `breakdown jsonb`, `inputs jsonb`). Snapshotted nightly so trend lines persist across model tweaks.
- **Inspired by.** Triple Whale / Lifetimely revenue forecasting (lifecycle/finance research §9) + Mosaic FP&A (referenced as overkill at our scale, but the projection-from-pipeline pattern is right). Implementation is in-app SQL not a vendor install — the model is forty lines and the data is already in Supabase.

#### 4.11.3 Real-time production projection

A live forecast of unit availability vs demand pull, refreshed on every Build / Fulfillment / Stock event.

- **Inputs:** `units` current state (ready / in-production / reserved / shipped / quarantine / scrap) joined to `batches.arrived_at` (expected arrival for in-production batches), reservation rate from the fulfillment queue, demand from `orders` with `kind='replacement'` and `awaiting_batch_id` set (the Kristen / Cheryl pattern), demand from open `orders` with `status IN ('pending','confirmed')`.
- **Model.** Per-batch burn-down: starting from today's `ready_count`, subtract weekly-reservation-velocity, add `arrived_at` step-functions when a future batch lands. Output the date each batch hits zero, then the date each SKU hits zero across all batches.
- **UI.** A `ProductionProjectionPanel.tsx` showing: per-batch burn-down chart (one line per batch — P150 / P50N / P100 / P100X — with the "in-production" lines starting at their projected `arrived_at`); a stock-out forecast table with columns `batch`, `current_ready`, `weekly_velocity`, `projected_stockout_date`, `replacement_queue_size`, `risk` chip (green = stock-out > 90d, amber = 30–90d, red = < 30d). Flags scenarios where projected demand crosses projected supply before the next batch lands — exactly the P100X situation that surfaced with Kristen + Cheryl awaiting end-of-July.
- **Data model.** New `production_projection_snapshots` (`id`, `as_of`, `batch_id` FK, `ready_count`, `reserved_count`, `weekly_velocity`, `projected_stockout_date`, `inbound_units`, `inbound_arrival_date`, `replacement_queue_size`, `risk_level`).
- **Sequencing note.** Naturally pairs with backlog #71's `awaiting_batch_id` (shipped) and the `freight_quotes` history table (proposal backlog #11) — both feed accurate demand-side signals.
- **Inspired by.** Demand Planning module in NetSuite / Cin7 (ERP research §1, §2) — pulled the projection pattern, not the vendor.

**Visibility model.** All three sub-panels live behind the same RBAC gate. George and Huayi see the full module; everyone else has no "Finance" tab in their nav and gets a 403 if they URL-hop to `/finance`. Activity-log entries written by this module include the actor so audit is preserved even though the surface is private.

**Touches.** New [modules/Finance](app/src/modules/Finance) directory with `index.tsx`, `JournalPanel.tsx`, `SalesProjectionPanel.tsx`, `ProductionProjectionPanel.tsx`; new [lib/finance.ts](app/src/lib/finance.ts), [lib/salesProjection.ts](app/src/lib/salesProjection.ts), [lib/productionProjection.ts](app/src/lib/productionProjection.ts); migrations `sales_projection_snapshots`, `production_projection_snapshots`, plus the QBO daily-summary tables from §6.15; route guard in [App.tsx](app/src/App.tsx); GlobalNav conditional render.

**Effort.** L overall. JournalPanel is M (already scoped in §6.15). SalesProjectionPanel + ProductionProjectionPanel are each M. Sequencing inside the module: ship JournalPanel first (proposal P1 #5), ProductionProjectionPanel second (depends only on existing Stock + Build data + #71), SalesProjectionPanel third (depends on Facebook Marketing API §6.4 for the pipeline-driven forecast).

## 5. Pre-sale marketing & campaigns — the missing domain

Today the lifecycle from "first ad impression" to "first order shipped" is invisible inside makeLILA. The product surface area to add:

**Campaign performance dashboard.** A new `modules/Marketing/CampaignsTab.tsx` aggregating Facebook Ads spend + reach + clicks + conversions, Klaviyo campaign opens + clicks + conversions, HubSpot sequence touches (or replacement if decommissioned). Tied to internal `orders` via UTM-tag + email-match resolution.

**Lead attribution per customer.** Add `customers.first_touch_source`, `customers.first_touch_campaign_id`, `customers.first_touch_at`, `customers.last_touch_source`, `customers.last_touch_campaign_id` populated at customer-create time from UTM tags on the originating Shopify order or HubSpot contact. Surfaced as a chip on JourneyTab so Pedrum sees "originated from FB-2026-Spring-P100" at a glance.

**Cost-per-acquisition view.** Joins `customers.first_touch_campaign_id` to Facebook Ads spend (via Meta Marketing API) and Klaviyo campaign cost. Powers a contribution-margin-by-channel report for Pedrum's weekly review.

**Conversion API (CAPI) server-side events.** Facebook's iOS-14-era requirement: client-side Pixel events lose accuracy due to ATT opt-outs and ad blockers. The 2026 best practice is server-side CAPI events fired from the backend on order creation, lead form submission, and key lifecycle moments. makeLILA already has the events (via `logAction`); a thin edge function reformats them for CAPI.

## 6. Integration strategy

Each integration is classified by **role** (System of Record / Input / Event Sink / Bidirectional / Auth) and **status** (Live / Partial / Planned / Decision Pending).

### 6.1 Shopify — Input + Event Sink (Live)

**Role:** Authoritative for ecommerce checkout. Inputs into makeLILA: orders, customers, line items, payment summary (`subtotal_usd`, `tax_usd`, `discount_total_usd`, `payment_methods`, `financial_status`), refunds. Event sink: fulfillment push (mark shipped + tracking via `push-shopify-fulfillments` edge function); $0 draft replacement orders.

**Existing edge functions:** `sync-shopify-orders`, `push-shopify-fulfillments`, `sync-shopify-customers`.

**Planned:** Backlog #6 bidirectional sync remains explicitly limited to "always-safe fields" (payment summary, financial status). Per [system-of-record.md](system-of-record.md), Shopify never overwrites operator-curated address / line-item data after initial import. Backlog #77 — extend `sync-shopify-orders` window backward to pre-#1070 history so 97 unmatched shipping rows resolve.

### 6.2 HubSpot — Input (Live) — **Decommission as CRM under consideration**

**Current role.** Lead capture (forms on lila.vip), sales sequences, contact CRM. Inputs into makeLILA: customer profile data via `sync-customers-from-hubspot` edge function (manual + on-demand).

**Decision framework — keep vs decommission as CRM.**

Reasons to **keep**:
- Pedrum's existing sales sequences and template library live in HubSpot.
- HubSpot forms on lila.vip are the lead-capture surface; replacing them is a website project, not a makeLILA project.
- HubSpot's free CRM tier costs nothing today; switching incurs migration cost.

Reasons to **decommission**:
- HubSpot's CRM features are duplicative of makeLILA's [Customers](app/src/modules/Customers) module post-sale, creating a "two systems of record" risk for post-purchase customers (the documented source-of-truth boundary in [system-of-record.md](system-of-record.md) was added specifically to manage this).
- Sales sequences for one-time hardware DTC are lightweight enough to handle via Klaviyo (which is more powerful for ecommerce-specific flows) plus Facebook lead-ad auto-replies.
- HubSpot pricing escalates sharply if more than two users need active sequences ($90+/mo per sales seat at Sales Hub Starter).
- Free Klaviyo + Meta Lead Ads + makeLILA covers the entire functional area for one-time hardware.

**Recommended decision posture (this PRD's stance).** Keep HubSpot for pre-sale lead capture forms on lila.vip for at least Q3 2026. **Decommission HubSpot's CRM and sequence layer** when:
1. Klaviyo Track API firehose (proposal P1 #1) is shipped, so makeLILA can fire all post-sale events directly to Klaviyo for sequence triggering.
2. Pre-sale lead nurture sequences can be rebuilt in Klaviyo (one-week migration project).
3. Pedrum confirms his sales sequence workflow can move to Klaviyo without losing functionality.

**Migration scope if decommission proceeds:**
- Export HubSpot contacts → upsert into `customers` with `lead_source='hubspot_legacy'`.
- Rebuild active sequences as Klaviyo flows triggered by `customer_created` event from makeLILA.
- Repoint lila.vip lead forms to either makeLILA's public form (preferred) or directly to Klaviyo.
- Retire `sync-customers-from-hubspot`; retain export bundle for audit.
- Estimated effort: M (1-2 weeks calendar).

### 6.3 Klaviyo — Event Sink + Predictive Source (Partial)

**Current role.** Email broadcasts triggered by Shopify events (Klaviyo's built-in Shopify integration). makeLILA does not push events to Klaviyo today.

**Planned (proposal P1 #1).** Klaviyo Track API firehose: every internal state change (`unit_shipped`, `unit_delivered`, `service_ticket_opened`, `service_ticket_resolved`, `replacement_shipped`, `refund_approved`, `journey_stage_changed`, `telemetry_status_changed`) fires to Klaviyo via a `klaviyo-track` edge function. `customers.klaviyo_profile_id` populated lazily.

**Planned (proposal P2 #10).** Nightly `klaviyo-pull-predictive` reverses the flow: pull Klaviyo's predicted CLV, expected next order date, and churn risk back to `customers` columns for operator-visible chips and sort/filter.

**SMS surface.** Klaviyo SMS is an open question — Quo currently owns operator-driven 1-on-1 SMS. Recommendation: keep Quo for 1-on-1, use Klaviyo SMS for broadcast/sequence-driven outbound (post-purchase Day 1 tip, Day 7 troubleshooting nudge).

### 6.4 Facebook (Meta) — Input + Event Sink (Planned — NEW)

**Why Facebook integration matters for makeLILA.** Facebook Ads is currently the largest single pre-sale marketing channel for VCycene. None of the spend, performance, or lead-flow data reaches makeLILA today, which means Pedrum cannot answer "what's our CAC by campaign" without leaving the app.

**Four sub-integrations to scope:**

**(a) Meta Marketing API — campaign spend + performance pull.** Daily edge function pulls campaign-level spend, impressions, clicks, conversions into a new `facebook_campaign_metrics` table. Powers the marketing dashboard (§5). Auth via Meta Business System User access token. Effort: M.

**(b) Meta Lead Ads — lead capture webhook.** Lead form submissions on Facebook/Instagram Lead Ads fire to a `meta-lead-webhook` edge function, which creates a `customers` row tagged `lead_source='facebook_lead_ad'` with `first_touch_campaign_id` populated. Optionally fires a Klaviyo event to start the lead-nurture flow. Effort: S.

**(c) Conversion API (CAPI) — server-side conversion events.** Outbound: edge function fires `Purchase`, `Lead`, `AddToCart`, `InitiateCheckout` events to Meta CAPI from makeLILA's internal events, with deduplication keys matching the Pixel events fired client-side. This is the 2026 best practice for iOS-14-era attribution recovery. Effort: M.

**(d) Catalog / Product Feed.** Sync `units` / SKUs to Meta Commerce Catalog for dynamic product ads. Only relevant if VCycene runs DPA campaigns; can be deferred. Effort: S.

**Recommended phasing:** (a) + (b) in Q3 2026 to close the visibility gap; (c) when ad spend justifies the engineering cost (likely Q4 2026); (d) deferred until DPAs are confirmed in the marketing plan.

### 6.14 Freightcom (via ClickShip) — Input + Event Sink (Live, Partial automation)

**Role.** Authoritative carrier-rate + label provider for outbound shipments, primarily Canadian LTL. Freightcom is the negotiated-rate aggregator (customer-of-record with the carriers); ClickShip is the Shopify-friendly UI layered on top of Freightcom's API. makeLILA inputs from Freightcom: per-order freight quotes used by [OrderReview/detail/FreightCard](app/src/modules/OrderReview), surfaced as `orders.freight_estimate_usd` + `orders.freight_estimate_source`. Event sink: label print + tracking-number push when the Fulfillment queue advances to the label step (carrier enum already supports UPS / FedEx / Purolator / Canada Post / Canpar / GLS — the actual Freightcom rate-shop result).

**Current state.** Quotes pulled at the order-card surface and on operator demand; selected rate written to `orders.freight_estimate_usd`. Each re-quote overwrites the previous value — there is no per-order quote history today.

**Planned (proposal backlog #11, P2).** Persist every quote into a new `freight_quotes` child table (`order_id` FK, `provider` enum `clickship` / `freightcom`, `service_level`, `rate_cad`, `rate_usd`, `transit_days`, `quoted_at`, `selected bool`, `raw jsonb`). Wrap the ClickShip and Freightcom calls behind a new [lib/freight.ts](app/src/lib/freight.ts) so every quote inserts a row. "Re-quote" never overwrites; the latest `selected=true` powers the order-row chip. Effort: S.

**Strategic note.** Freightcom's lock-in is the negotiated CA carrier rate sheet — switching to a self-managed rate-shop (EasyPost / Shippo, §7) sacrifices that discount unless makeLILA holds its own carrier accounts. The recommended posture is **dual-call**: keep ClickShip / Freightcom for CA LTL where the discount applies, evaluate EasyPost only for US parcel where bring-your-own-account economics win.

### 6.15 QuickBooks Online — Event Sink (Planned, P1)

**Current state.** Not integrated. Reconciliation against Shopify, Sezzle, ClickShip, and the bank happens manually each Monday in QBO's web UI. Per-order Shopify-to-QBO push is explicitly **not** the chosen direction (see §10 out-of-scope) — at the makeLILA volume and the multi-gateway / multi-currency / partial-refund reality, per-order detail breaks reconciliation; daily summary journals are the industry standard (Synder / A2X pattern). Per [system-of-record.md](system-of-record.md), QBO is downstream of makeLILA for revenue / refund / COGS data; makeLILA never reads back from QBO.

**Planned (proposal P1 #5).** Build a `qbo-daily-summary` Supabase scheduled edge function that, nightly:

1. Aggregates the day's `orders`, `returns` / refunds, and Sezzle-payout rows from makeLILA grouped by `(currency, payment_channel)`.
2. Posts **one journal entry per group** to QBO via OAuth 2.0 with line items mapped: gross sales → revenue GL, discounts → discount GL, refunds → refund GL, tax → tax-liability GL, shipping → shipping income GL, fees → merchant-fees GL, net → bank/clearing GL.
3. Writes the QBO journal id back to a new `qbo_daily_journals` table (`date`, `currency`, `payment_channel`, `gross_sales`, `discounts`, `refunds`, `tax_collected`, `shipping`, `fees`, `net_deposit`, `qbo_journal_id`, `posted_at`).
4. Replacement orders (the $0 draft-order pattern documented in §6.1 + competitive proposal §1) are excluded from revenue and routed to a `warranty_reserve` GL account so they do not double-count revenue.

A new `modules/PostShipment/FinanceTab.tsx` surfaces the last 30 days with **Repost** (idempotent on journal id) and **View in QBO** buttons; daily summaries that fail to post are surfaced as exception rows for George.

**Why now.** Eliminates George's Monday reconciliation work; closes the silent-revenue-leakage class of risk identified in the risk-first synthesis (per-currency ledger errors + partial-refund miscalc + double-counted replacement orders). Effort: M.

**Sequencing.** Depends on the active Returns/Refunds overhaul (proposal P1 #4) shipping first so refund disposition data is trustworthy before it hits the GL.

**Auth + ops.** OAuth 2.0 with QBO; refresh tokens stored in Supabase Vault. 500 req/min / 10 concurrent ceiling per QBO company is far above our daily-summary cadence. Token expiry (60 min) handled by the edge function with a refresh-and-retry wrapper.

**Explicitly NOT in scope for this integration.** Per-order push (rejected per §10); reading back from QBO into makeLILA (one-way only); installing Synder / A2X / Bold Commerce (the aggregation SQL is ~40 lines and we want the rules in the repo per competitive proposal §1); Avalara / Vertex (Stripe Tax sufficient up to ~$5M ARR per §7).

P1 V1 lift: viewport meta + safe-area-insets + PWA manifest + AppShell narrow-aware. P2: per-module table → card-row treatment under ~700px.

**Observability.** Every mutation writes `activity_log` via `logAction()`. KPI panel surfaces volumes per action type. Proposed extension (proposal P2 #12): add `entity_type` / `entity_id` / `unit_serial` columns + per-serial timeline UI.

**Backup + audit.** Supabase nightly backups. Activity log is the audit trail. No separate DR plan currently; revisit if a hard SLA is required.

## 10. Explicit out-of-scope

- Replacing makeLILA with any incumbent ERP/CRM (NetSuite, SAP B1, Salesforce, Zendesk, Katana, Cin7). At 50-200 units/mo the licensing + implementation cost exceeds the entire ops headcount budget.
- Salesforce Field Service Lightning or Aquant. Built for fleets of dispatched technicians; VCycene is depot-only.
- Full Tulip / 21 CFR Part 11 MES compliance. Non-regulated consumer hardware.
- Native iOS / Android apps. PWA installable to home screen is sufficient.
- Offline-first / sync-when-online. Operator UX assumes a live Supabase connection.
- In-app workflow editor (n8n-style GUI in app). Klaviyo + Supabase scheduled functions cover all 90-day automation; n8n self-host (§7) is the right tier when complexity grows.
- Conversation-merging omnichannel inbox across Gmail + Quo + Resend. Quarter-long rewrite of the messaging spine for marginal triage benefit over the per-serial timeline + device-context header.
- Per-order Shopify-to-QBO sync. Daily-summary journal (proposal P1 #5) gives better grouping control at zero vendor cost.
- Homegrown weighted customer health score. Klaviyo's predictive properties (proposal P2 #10) are trained on richer data.
