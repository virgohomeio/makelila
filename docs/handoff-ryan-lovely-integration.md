### 1. Shared secret ✅ DEPLOYED

`X-Lovely-Secret` header auth on the ingest endpoint. The secret was generated by Huayi as 64 hex chars (256 bits) and stored in three places:

- `LOVELY_INGEST_SECRET` env var on makelila Supabase (txeftbbzeflequvrmjjr) — read by the edge function
- `MAKELILA_INGEST_SECRET` env var on lovely Supabase (arfdopgbvlfmhmcfghhl) — for parity / future operator-fires-push reverse direction
- `public.app_config` row keyed `'makelila_ingest_secret'` on lovely Supabase — read by the pg_notify triggers (since hosted Supabase doesn't allow `alter database ... set` for custom GUCs)

**To rotate:** generate a new value, update LOVELY_INGEST_SECRET on makelila first (recipient ready to accept either), then update the app_config row on lovely:

```sql
update public.app_config set value = '<new>', updated_at = now()
 where key = 'makelila_ingest_secret';
```

Stored long-term in 1Password (or the VCycene Notion access page) per Huayi.

### 2. Webhook payload contract

POST to `https://txeftbbzeflequvrmjjr.functions.supabase.co/ingest-lovely-event` with:

```http
POST /ingest-lovely-event
Content-Type: application/json
X-Lovely-Secret: <the-shared-secret>

{
  "event_type": "lovely.onboarding_step",
  "lovely_user_id": "auth-users-uuid-here",
  "lovely_email": "ryan@example.com",     // optional but always send if you have it
  "serial_number": "LL01-00000000123",    // optional; sent post-pairing
  "occurred_at": "2026-06-08T01:23:45Z",  // optional; defaults to now()
  "payload": {                             // optional event-specific data
    "from": "welcome_done",
    "to": "quiz_done"
  }
}
```

The ingest function responds with:
```json
{ "ok": true, "resolution": "serial" | "email" | "unresolved",
  "customer_id": "uuid-or-null", "event_type": "lovely.onboarding_step" }
```

`200` = accepted (even if `resolution: "unresolved"` — those events are still saved and can be back-resolved when the user pairs).
`400` = bad payload (missing event_type or lovely_user_id).
`401` = bad secret.
`500` = transient DB error — safe to retry.

| `lovely.push_opt_in` | new `push_subscriptions` row | `{}` | trigger on `push_subscriptions` |

### 4. Emit mechanism ✅ DEPLOYED — server-side triggers

Full SQL snapshot of what's live: [docs/lovely-side-deployed-2026-06-07.sql](lovely-side-deployed-2026-06-07.sql).

Summary of what got deployed to lovely Supabase:

- `pg_net` extension enabled
- `public.app_config` table (no RLS policies → service_role + SECURITY DEFINER only)
- `public.fire_makelila_event(user_id, event_type, serial, email, payload)` helper — reads secret from `app_config`, POSTs to makelila ingest endpoint asynchronously, swallows all exceptions so it can NEVER block the parent insert/update
- AFTER INSERT trigger on `public.users` → `lovely.signup`
- AFTER UPDATE trigger on `public.users` → fires `lovely.serial_paired` (on first non-null serial_number), `lovely.onboarding_step` (every step change), and `lovely.onboarding_done` (when step becomes `tour_done`)
- AFTER INSERT trigger on `public.damage_reports` → `lovely.damage_report` (with notes_present + photo_count derived from images table)
- AFTER INSERT trigger on `public.ota_acceptances` → `lovely.ota_accepted` (with resolved ota_version)
- AFTER INSERT trigger on `public.push_subscriptions` → `lovely.push_opt_in`

The earlier `current_setting('app.makelila_ingest_secret', true)` approach in the original draft of this doc was abandoned — hosted Supabase rejects `ALTER DATABASE postgres SET <key>` with `42501: permission denied`. The `app_config` table + `select value from ...` lookup inside the SECURITY DEFINER helper is the working pattern.

**Client-side POST (for engagement-only events).** For `lovely.dashboard_open` and `lovely.batch_complete_seen`, the client fetches:

```ts
// in app/dashboard/page.tsx (or a layout)
useEffect(() => {
  const last = localStorage.getItem('mk_dashboard_open_at');
  const now = Date.now();
  if (last && now - Number(last) < 10 * 60 * 1000) return; // debounce 10min
  localStorage.setItem('mk_dashboard_open_at', String(now));
  fetch('/api/lovely-event', {
    method: 'POST',
    body: JSON.stringify({ event_type: 'lovely.dashboard_open' }),
  });
}, []);
```

Then a thin `/api/lovely-event` Next.js route reads the user session, attaches `lovely_user_id` + `email` + `serial_number`, and POSTs to the makelila ingest endpoint with the secret (which lives server-side in lovely's env).

### 5. Testing — ✅ end-to-end verified 2026-06-08 02:06 UTC

Two smoke fires from `fire_makelila_event` via the lovely SQL editor:

1. **Unresolved path** — fake UUID + `huayi@virgohome.io` (operator email, not in `customers`). Result: `customer_id: null`, `resolution: 'unresolved'`, event row landed in `customer_events`. ~700ms round trip (occurred_at → ingested_at).
2. **Email-resolution path** — fake UUID + a real customer's email (Sherry Tang). Result: `customer_id: 7647c2a7-...`, `resolution: 'email'`, link row created in `customer_app_links` with `resolution='email'`. Event correctly attached to the real customer.

Both rows cleaned up after the test so Sherry's real timeline isn't polluted.

The third resolution path (`serial → units.customer_id`) wasn't smoke-tested but uses the same code branch — first real onboarding paring event from a lovely user will validate it.

To re-run a smoke test yourself, from lovely's SQL editor:

```sql
select public.fire_makelila_event(
  gen_random_uuid(),                       -- pretend lovely user
  'lovely.test',                            -- event_type
  null,                                     -- serial_number (or a real LL01-... to test serial path)
  'sherrytang98@gmail.com',                 -- email (use any real customer email to test email path)
  '{"smoke_test": true}'::jsonb
);
```

Then on makelila side (via MCP / Supabase SQL editor):
```sql
select * from public.customer_events order by ingested_at desc limit 5;
```

Clean up afterwards so test rows don't pollute the timeline:
```sql
delete from public.customer_events where event_type = 'lovely.test';
delete from public.customer_app_links where resolution = 'unresolved' or lovely_user_id in (<test uuids>);
```
