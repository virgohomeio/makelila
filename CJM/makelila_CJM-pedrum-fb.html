Goals & Motivations

  • Replace 5+ scattered spreadsheets with one trusted app
  • Walk every order through Assign → Test → Label → Dock → Email cleanly
  • Know exactly which serial is where, in what state, for which customer
  • Catch the P150 35% defect rate before it reaches customers
  • Stop briefing colleagues over Slack DMs and free-text cells

Struggles & Frustrations

  • Fulfillment was split across CA / US / Personal / Hold / Returns tabs
  • Replacements buried in Notes columns; no structured field
  • Customer follow-up calendar lived in a separate sheet (Ash)
  • Rework requests came as text messages to Junaid
  • No SO → campaign attribution (Pedrum)
  • Two users editing the file = corrupted state
  • Status changes had no audit trail — "who changed this and when?"

Context

Inventory: 350 units produced across 5 batches (P50, P150, P50N, P100, P100X). Heavy serial-number tracking (LL01-XXXXXXXXXXX). Active rework backlog from P150. ~20 P100 replacements queued. Real customers, real money — VCycene has refunded ~$16K across 32 returns.

Outcome

makelila becomes the system of record for every LILA unit, every order, and every team handoff — and the spreadsheets become read-only history.

🚀 Opportunities — makelila

Awareness

Make the migration visible — kill one tab at a time

Each shipped module retires a specific spreadsheet tab. Publish a Slack-pinned scoreboard: "Order Review live → Shopify orders import tab retired"; "Fulfillment Queue live → CA/US/Personal/Hold/Returns tabs retired." Adoption is faster when teammates see real wins, not roadmap promises. (Source: Ashwini audit · architecture.md)

Onboarding

Identity dropdown is fine for now — but ship roles before P200

The "I am" localStorage picker works for 7 people who all see the same screens. The moment external partners (factory, freight) need limited views — or finance needs cost-only access — the RLS roles defined in makelila-architecture.md §7 become unblocking. Plan to ship role gates before headcount or partner integrations grow. (Source: app-design §10 open question · architecture §7)

Order Review

Surface campaign attribution at triage time

Pedrum's gap — no SO → campaign attribution — is solved by the campaigns table and orders.campaign_id FK. Surface the source campaign as a chip on the OrderRow so reviewers immediately see "Holiday Sale 2025 · 12mo Sezzle" and can flag low-quality acquisition before fulfillment. Closes the loop on returns reasons too. (Source: Ashwini gap #5/#7 · Pedrum)

Shelf

Print-ready shelf snapshot for the warehouse wall

Aaron and Raymond still print the layout. Add a "Print shelf" button that exports an A3 PDF of the current grid with serial last-5, batch chip, and status colour. One click. Eliminates the hand-drawn layout sheet that walks the floor today. (Source: Aaron · Raymond)

Fulfillment Queue

Hard-block Step 5 send when starter-kit field is empty (US only)

The single highest-cost bug in fulfillment is shipping a US LILA Pro without the starter kit. The Step 5 design already requires starter_tracking_num for US orders, but the validation must live both in the UI and in the send-fulfillment-email edge function — never trust the client. Add a 502 path with a clear error banner. (Source: fulfillment-design §6 step 5)

Fulfillment Queue

Resend webhook → email_delivery_status table

Right now the queue marks email_sent_at on Resend's 2xx response, but doesn't track delivery, opens, or bounces. Subscribe to Resend webhook events into a email_delivery_status table; surface bounces as an Activity Log alert. Bounced shipping confirmations to wrong customer email are how disputes start. (Source: AI suggested)

Post-Shipment

Cost-per-km outlier flag — drives carrier renegotiation

The Post-Shipment dashboard already computes cost/km. Flag the top-decile outliers in red (e.g. remote BC, rural US states) and feed a monthly "renegotiation candidates" view. Currently those costs hide as line items; surfacing them is leverage at carrier-contract renewal. (Source: psShipments dataset)

Stock

Bulk-import flow for incoming batches (P100X next)

P100 was seeded via a SQL migration. P100X arrives in ~Sep 2026 and shouldn't require a developer. Build a CSV-upload flow on Stock → Add that ingests serial range + batch + manufacture date and creates stockData rows in ordered status. Required before next batch arrival. (Source: app-design §12 open question · inventory-status)

Stock

Add unit_status_history table for full lineage

The Update button pattern commits status changes but only writes the latest state to stockData. The architecture doc lists unit_status_history (unit_id, from_status, to_status, user, ts) as inferred — promote it to a real migration. This is the only way to answer "when did LL01-…0354 go from ca-test to ready" 6 months from now. (Source: makelila-architecture §10)

Rework / Returns

Defect-to-campaign attribution loop

P150's 35% defect rate is known internally but not visible at marketing review. Tag every unit_reworks row + every return_process with the originating campaign_id via the SO chain. Pedrum's marketing review can then see "this campaign's customers had 2× the return rate" and decide whether to repeat it. (Source: Ashwini gap #5/#7 · Junaid)

Rework / Returns

Parts BOM on repair_tickets — drives reorder POs

Junaid's rework requests should each carry a parts_needed field that aggregates into a "parts to reorder" rollup. Top Lid replacements are 5 of 9 — that signal should auto-trigger a draft PO when the in-stock spare count hits the reorder point. (Source: Junaid · Ashwini gap #6)

Activity Log
