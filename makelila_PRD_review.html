§1 Executive Summary

Overview

makeLILA is the unified internal operations platform for VCycene's LILA Composter business. It replaces what the market would otherwise stitch together as five separate SaaS subscriptions — a CRM, a customer service desk, a lightweight ERP, a finance / lifecycle marketing stack, and a pre-sale campaign manager — with one React + TypeScript + Supabase application owned by the team and built around hardware-product specifics that off-the-shelf tools do not model natively.

Strategic posture

Per docs/system-of-record.md: makeLILA is the system of record for internal operational data; every external platform is either an input that seeds records or an event sink that receives state changes. No external system should ever overwrite operator-curated data after the initial sync. This posture drives every integration decision in §6 and §7.

§9 Architecture & non-functional requirements

Architecture & NFRs

Stack (live)

Frontend: React 18 + TypeScript + Vite + CSS Modules.
Backend: Supabase (Postgres + Auth + Realtime + Storage + Edge Functions).
Testing: Vitest (unit) + Playwright (e2e).
Hosting: GitHub Pages (virgohomeio/makelila) via deploy.yml workflow.
Auth: Supabase Google OAuth restricted to @virgohome.io.

Two-Supabase-project posture

Main project txeftbbzeflequvrmjjr (LILA-Pro-Inventory): operational system of record. All ops modules read/write here.
Telemetry project arfdopgbvlfmhmcfghhl ("Lovely"): hosts device telemetry — events, ac_current, bme_sensors, temperature_sensors, machine_health, lila. Dashboard reads via lib/supabaseTelemetry.ts; read-only with separate auth.

Non-functional requirements

Performance: Realtime subscriptions on hot tables (orders, service_tickets, units, fulfillment_queue, ticket_notes, service_ticket_attachments). Pagination not currently required at 50–200 orders/mo + ~1000 lifetime units; revisit at 5× volume.

Security: Supabase RLS in place, currently permissive for authenticated @virgohome.io users. P1 hardening: profiles.role enum (operator / manager / finance / admin) seeded from MANAGER_EMAILS / FINANCE_EMAILS; central canDo(role, action) for write gating; canView(role, module) for module-level visibility (drives Finance module nav-hide + route guard + RLS). Finance role = George + Huayi.

Mobile: Backlog #80 — app today is desktop-only despite installable to iPhone home screen. P1 V1: viewport meta + safe-area-insets + PWA manifest + AppShell narrow-aware. P2 V2: per-module table → card-row treatment under ~700px.

Observability: Every mutation writes activity_log via logAction(). KPI panel surfaces volumes per action type. P2: entity_type / entity_id / unit_serial columns + per-serial timeline UI; this also unblocks the OKR/KPI tracking in §4.9.

Backup + audit: Supabase nightly backups. Activity log = audit trail.
