§1 Executive Summary

Overview

makeLILA is the unified internal operations platform for VCycene's LILA Composter business. It replaces what the market would otherwise stitch together as five separate SaaS subscriptions — a CRM, a customer service desk, a lightweight ERP, a finance / lifecycle marketing stack, and a pre-sale campaign manager — with one React + TypeScript + Supabase application owned by the team and built around hardware-product specifics that off-the-shelf tools do not model natively.

Strategic posture

Per docs/system-of-record.md: makeLILA is the system of record for internal operational data; every external platform is either an input that seeds records or an event sink that receives state changes. No external system should ever overwrite operator-curated data after the initial sync. This posture drives every integration decision in §6 and §7.

§9 Architecture & non-functional requirements

Architecture & NFRs

Stack (live)

Frontend: React 18 + TypeScript + Vite + CSS Modules.
Backend: Supabase (Postgres + Auth + Realtime + Storage + Edge Functions).
Testing: Vitest (unit) + Playwright (e2e).
Hosting: GitHub Pages (virgohomeio/makelila) via deploy.yml workflow.
Auth: Supabase Google OAuth restricted to @virgohome.io.

Two-Supabase-project posture

Main project txeftbbzeflequvrmjjr (LILA-Pro-Inventory): operational system of record. All ops modules read/write here.
Lovely project arfdopgbvlfmhmcfghhl: hosts device telemetry (events, ac_current, bme_sensors, temperature_sensors, machine_health, lila — read by Dashboard via lib/supabaseTelemetry.ts) AND, as of v1.5, the lilalovely customer-facing PWA owned by Ryan Yuan (own auth.users + public.users + compost_batches + damage_reports + ota_acceptances + push_subscriptions + app_config). Same Postgres, separate concern.

Cross-project edge function pattern (v1.5): lovely-side pg triggers call fire_makelila_event() SECURITY DEFINER helper that reads a shared secret from public.app_config (hosted Supabase blocks ALTER DATABASE ... SET for custom GUCs) and POSTs to makelila's ingest-lovely-event edge function with X-Lovely-Secret header. Same posture will be reused for makelila → lovely reverse direction (operator-fires-push) — POST to /api/admin/push with the same shared secret.

New tables (v1.5): customer_events (append-only event stream, indexed by (customer_id, occurred_at desc)) · customer_app_links (lovely_user_id ↔ customers.id via resolution = 'serial' | 'email' | 'unresolved') · customer_engagement_summary view (dormancy_days + events_7d + events_30d roll-up) · external_email_allowlist (non-@virgohome.io users opt-in via auth.tsx EXTERNAL_ALLOWLIST + patched handle_new_user trigger) · lovely-side app_config (key/value with no RLS, service_role + SECURITY DEFINER only).

          Calibration notes

          Repo cadence (2026-04-01 to 2026-06-06): 430 total commits, 158 of which are feat(*) commits. Peak day: 2026-06-04 with 80 commits. Service module shipped its first functional surface ~2 weeks after design spec; Build module ~3 weeks; Service replacement workflow (#55) was specified and shipped within one week. The original 24-week timeline assumed conservative software-industry pace (1 PR / week / engineer with weekly code review). The 14-week revised timeline uses the actual observed cadence.

          Token estimates are based on sampling the size of recent commits (file changes × ~30–80K output tokens per commit) plus context overhead. Input multiplier ranges 5–10× depending on how much codebase context is loaded per session; we use the upper end (10×) for total estimates to stay conservative.

          Tier weekly limits are approximate. Claude's published 5-hour-window limits multiplied by typical session output. Your actual mileage will depend on session length, model mix, and how heavily Workflow / Agent tools are used (those multiply token consumption ~3–7× via subagent fan-out).
